The Ask

Clients want their results as a spreadsheet

  • Platform is building client portal screens that show clients their audit information, including the medical coding being reviewed.
  • Client admins want to export that data to Excel and have it emailed to them.
  • The catch: the data lives in Audit, but the screens and the email live in Platform.
Audit · has the data

Where the work already lives

  • Holds every released evaluation and its scores
  • Already renders spreadsheets via XLSX::Workbook
  • Knows which cycles a client is allowed to see
Platform · faces the client

Where the client already is

  • Owns the client portal and the Export button
  • Has the email infrastructure, ready to send
  • Has the sign-in logic that gates the download

The Approach

Background jobs on both sides

  • Both apps run Sidekiq, and both create a record the moment a request starts.
  • The user gets an immediate answer: the click returns a 202 and a friendly flash, never a spinner.
  • The slow part, building and moving the file, happens off to the side where a retry is cheap.
Audit generates & retains
  • Runs the visibility query over released evaluations
  • Builds the file through a client-safe column contract
  • Keeps the bytes in ActiveStorage and serves them on request
  • Stays the single authority over the file
Platform delivers
  • Triggers the export from the portal
  • Pulls the bytes back and verifies them
  • Sends the email through existing infrastructure
  • Gates the download behind sign in

Audit Branch

What was done in Audit

  • API controller takes the request, returns 202, kicks off a job.
  • Generation job runs the query, renders the workbook, attaches it, fires the callback.
  • Visibility query and column contract keep the export scoped and client safe.

Audit · API Controller

app/controllers/api/v1/audit_exports_controller.rb 
def create
  export = AuditExport.create!(client: @client, platform_export_id:, scope:, ...)
  AuditExports::GenerationJob.perform_async(export.id)
  render json: {audit_export_id: export.audit_export_id, status: "accepted"}, status: :accepted
end

def show
  export = AuditExport.completed.find_by(audit_export_id: params[:id])
  return head :not_found unless export&.file&.attached?
  send_data export.file.download, filename: export.filename, type: export.content_type
end

Platform Branch

What was done in Platform

  • Request worker submits to Audit and stores the returned id.
  • Webhooks controller receives the ready callback and enqueues ingest.
  • Ingest worker and mailer fetch, verify, attach, then email the link.

Platform · Ingest Worker

app/workers/audit_export_ingest_worker.rb 
def perform(export_id)
  export = AuditExport.find_by(id: export_id)
  return if export.completed? || export.failed?

  fetch_verify_and_attach(export)   # GET bytes, verify sha256, attach
  export.mark_completed!
  deliver_email(export)             # gated by delivered_at, sent once
end

Section Three

The webhook layer

The one genuinely new capability, kept generic on both sides

Audit · outbound

A sender that knows nothing

  • Webhooks::Sender takes a url, token, payload
  • Webhooks::DeliveryJob retries only transient failures
  • No domain knowledge, reusable for any callback
Platform · inbound

A receiver that just verifies

  • Constant-time bearer token check
  • Idempotent: replays of a finished export are ignored
  • Acks 204, hands the work to a job

Why Notify, Then Fetch

The callback carries news, not cargo

  • The webhook sends file metadata only, so no large payloads ride the callback.
  • Platform pulls the bytes from Audit's authenticated endpoint, so Audit stays the single authority over the file.
  • Those bytes are checksum verified before anyone is emailed.
