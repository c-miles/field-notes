AUD-791 / Audit · Platform

Web components, as a spike.

An evaluation of how Audit hands a slice of itself to Platform without giving up ownership of either the data or how it's drawn.

field notes   ·   engineering 2026
01 / Why this exists

The arrow has only ever pointed one way.

Until now, the Audit / Platform integration has run in a single direction. Platform sends us coders, charts, and team data. We ingest it, react, do audit work, and emit events. Platform does not read anything back.

This is the first time Platform needs to read a surface from us. The question on the table is the integration shape.

The Goal

A hard line, both ways

Audit owns the data. Audit owns how it's drawn. Platform owns only the page wrapped around it. No JSON ingestion, no parallel ERB, no mirrored data model.

02 / The Primitive

Custom elements are a browser native.

A web component is a class that extends HTMLElement and is registered to a tag name. The browser handles the rest. There is no framework, no runtime, no SPA. Three lifecycle moments matter for us.

  • constructor — the instance is allocated, the shadow root attaches
  • connectedCallback — the element joins the DOM, we kick off the first fetch
  • attributeChangedCallback — an observed attribute changed, refetch if needed

MDN: Using custom elements

register the tag js 
class AuditApprovalQueue extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: "open" });
  }
  connectedCallback() {
    this.loadQueue();
  }
}

customElements.define(
  "audit-approval-queue",
  AuditApprovalQueue
);
03 / The Artifact

One job per tag.

Each custom element renders one surface. Pages on Platform compose them. Navigation between surfaces is Platform's job, not the component's.

In this PR

<audit-approval‑queue>

Renders Audit's external approval queue. Read-only list of cycles awaiting Platform review. The spike happens to include an internal queue-to-detail view too, but that's a spike convenience, not the production shape.

In real life

A small fleet

Each surface gets its own tag. A queue component on the queue page, a separate detail component on the detail page, dispute review on its own page, and so on. Platform's router decides which page is which.

The contract

Events out, attrs in

Click View on a row, the queue dispatches cycle-selected with the id. Platform listens, navigates to its detail page, that page embeds the detail component with the id as an attribute. Components stay independent.

04 / Boot Sequence

What happens when the page loads.

Step 01
Platform renders its page
<script src> pointing at Audit, plus <audit-approval-queue> in the markup
→
Step 02
Browser fetches the bundle
audit.kodehealth.com/audit-components/v1/audit-approval-queue.js
Step 03
The IIFE runs immediately
customElements.define("audit-approval-queue", …)
→
Step 04
Every matching tag upgrades
constructor → attachShadow → connectedCallback
Step 05
connectedCallback → loadQueue()
fetch(/api/v1/components/approval_cycles), inject HTML fragment into the shadow root

Custom elements upgrade automatically, even if the script defines the class AFTER the tag is already in the document. Order does not matter.

05 / The Wall

Shadow DOM, open mode.

An isolated DOM subtree attached to our element. Platform's stylesheet can't reach in, Platform's JS can't querySelector us, and inherited properties like font-family still flow down from the host page.

What we chose

Open Shadow DOM

Our class names (c-card, c-btn) live in their own scope and won't collide with Platform's overlapping ones. Host code can still introspect via element.shadowRoot, which is useful for integration tests.

Considered

Closed Shadow DOM

Same isolation, but element.shadowRoot returns null externally. No real security benefit here (the bundle is public, attributes are readable), and it makes debugging and tests harder. Not worth the tradeoff.

Considered

Light DOM (no shadow)

Render straight into Platform's tree. Inherits Platform's CSS, which is great when you want host theming and bad when you have overlapping class names. Live option if we ever decide Platform should style us.

Future hook: CSS shadow parts (::part()) let us selectively expose stylable bits to Platform without giving up the wall. — MDN: Using shadow DOM

06 / Crossing the Wall

Events leave the shadow root only if you say so.

By default an event fired inside a shadow root does not escape to the host page. To let Platform's Stimulus controller hear us, we set composed: true.

bubbles: true alone only bubbles within the shadow root. composed: true is the magic flag that lets the event cross into the parent tree where Platform's listeners live.

Two events leave the wall: cycle-selected when a user clicks View on a row, cycle-cleared when they click Back. Platform listens, Platform reacts.

MDN: Event.composed

dispatch a composed event js 
this.dispatchEvent(
  new CustomEvent("cycle-selected", {
    detail: { cycleId },
    bubbles: true,
    composed: true  // crosses the boundary
  })
);
07 / The Bundle

One file. One request.

The component source imports a Tailwind CSS file as a string. esbuild's --loader:.css=text flag turns the import into a literal JS string at build time. The compiled CSS lives inside the JS bundle.

At runtime the component injects it as an inline <style> in the shadow root. No separate stylesheet to host, no second HTTP request, no version skew between the JS and the CSS that styles it.

This is the pattern Stripe Elements, Intercom, and Lit's static styles all use. The right shape for shipping a third-party embed.

the pipeline, in two steps sh 
# 1. Tailwind compiles a scoped CSS file
tailwindcss \
  -i app/assets/tailwind/audit_components.css \
  -o app/javascript/components/styles.css

# 2. esbuild bundles the JS and inlines the CSS
#    as a JS string constant
npx esbuild app/javascript/components/audit_approval_queue.js \
  --bundle --format=iife \
  --loader:.css=text \
  --outfile=public/audit-components/v1/audit-approval-queue.js
08 / Delivery

The bundle is a static file.

The compiled bundle lands in public/audit-components/v1/. Rails' static-file middleware serves it directly. No controller runs. No auth check fires. Same delivery model as a logo PNG or jQuery's CDN.

Anyone with the URL can fetch it. Right now nothing sensitive lives in the bundle, just the component class and the inlined CSS. No API keys, no user data, no secrets baked in.

If a future component ever needs to ship something we'd rather not hand out anonymously, the bundle stops being the right place for it. Access control belongs one layer down, on the data endpoints the bundle calls.

Worth keeping in mind

Public by default.

Anything that lands in public/ is fetchable from the open internet. That works for component code because the code is supposed to be readable. It would not work for credentials, user data, or business logic we'd want to keep proprietary.

09 / Versioning

Path-versioned. Short-cached.

The URL is mutable: /audit-components/v1/audit-approval-queue.js always points at the current build. We bump to /v2/ only when the external contract changes — tag name, attribute names, dispatched event names.

A Rack middleware caps Cache-Control at one minute with stale-while-revalidate for five. Rails' default year-long cache on the public directory would have left embedders stuck on the pre-deploy bundle.

A diagnostic version.json sidecar carries the git SHA so anyone can curl and know what's deployed.

Cache-Control response header http 
Cache-Control: public,
               max-age=60,
               stale-while-revalidate=300

Browsers serve the cached copy immediately and revalidate in the background. A fresh deploy reaches users within roughly one minute, with no latency penalty during the revalidation window.

10 / Versioning, paths not taken

Alternatives still on the table.

Likely future

Manifest endpoint

JSON that points at the current bundle URLs for each component. Pays off once we ship a fleet. Lets Platform load components by name without caring about path conventions. Probably the answer when this becomes three or more components.

Considered

Hash filenames

Cache-forever fingerprinted assets. Excellent caching but Platform would have to discover the current hash on every render. Better as a downstream chunk fetched by a small loader than as the entry point.

Considered

Semver in path

e.g. /1.2.3/embed.js. Right shape for shipping a library to many third-party consumers with release-train coordination. Overkill for one internal team.

Considered

npm package

Platform installs the component as a build-time dep, lockfile pins the version. Right end-state once the contract stabilizes. Wrong during active spike work, where every component tweak would become a publish + Platform deploy.

Considered

Separate repo

Conflates ownership boundary with versioning. Adds CI / Dependabot / review-cycle cost. None of the conditions that justify a split (multiple consumers, different release cadence, different team) currently hold.

What we chose, why

Mutable /v1/

Ships first, stays out of the way fastest, leaves a clean migration path to manifest + npm later. The 60-second cache window plus the version sidecar covers the staleness concern that drove the question.

11 / CORS, briefly

Two origins, one rule the browser refuses to forget.

A page from app.kodehealth.com cannot, by default, fetch() audit.kodehealth.com and read the response. That's the Same-Origin Policy. CORS is the protocol the target server uses to opt in.

Our component sets a custom Authorization header. Custom headers trigger a preflight: the browser sends an OPTIONS first to ask permission, then the real request if Audit said yes.

What we configured
  • Origin allowlist sourced from a single env var (CORS_ALLOWED_ORIGINS), per environment
  • Scoped to /api/v1/components/ only — the only browser-callable surface
  • Catch-all OPTIONS route inside that namespace handles all preflights
  • 24-hour max-age so preflights cache and repeat fetches skip the round trip

12 / Authorization

Platform mints. Audit verifies.

Platform mints a short-lived HS256 JWT for the current user at page render. The component sends it as Authorization: Bearer on every fetch. Audit verifies the signature, the issuer, the audience, and the expiry before acting on anything in the claim.

User identity and role both ride in the verified payload. The approve endpoint stamps external_reviewer_id from the verified sub rather than trusting a body field a caller could lie about.

  • Per-user, not per-app — Audit knows which person acted
  • Role rides in the claim — no cross-app user table lookup on Audit's side
  • Expires in 30 minutes — short window mitigates token-leak replay without needing a denylist
  • Audience-bound — the token is usable only for the named verifier
  • Rotates via a key id header — no JWKS endpoint needed for the symmetric case
13 / The JWT, up close

HS256. Same shape Shopify embedded apps use.

Both apps share an HS256 signing secret via Rails credentials. Platform signs, Audit verifies. No JWKS endpoint, no asymmetric ceremony, no IdP round trip. Two services under one company, same shared infrastructure.

Audit's verifier enforces an issuer allowlist, the audience claim, and exp (with 30 seconds of clock skew tolerance). No email or other PII rides in the payload.

verified payload jwt 
{
  "iss": "app.kodehealth.com",
  "aud": "audit.kodehealth.com",
  "sub": "42",
  "role": "system_admin",
  "roles": ["system_admin", "staff"],
  "exp": 1779310032,
  "iat": 1779308232,
  "jti": "e158827c-6b22-413e-..."
}
// header carries kid: "v1" for rotation
14 / Auth, paths not taken

Other shapes that were on the table.

Intercom HMAC

user_hash = HMAC(user_id, secret). The minimum-viable upgrade from a shared key. But no expiry, no audience, no per-user revocation. Intercom themselves now deprecate it for JWT, for the same reasons.

Stripe ephemeral keys

Three-tier publishable / secret / session-token hierarchy. Correct for untrusted third-party merchants. Cosplay for two same-company Rails apps.

OAuth Token Exchange

RFC 8693. The architecturally pure answer when both apps share an IdP. Right end state. Requires registering Audit's API as a Logto resource and putting access tokens in front of every Platform session first — not the right next step.

Migration path stays open: the wire format is JWT. Swapping a Platform-minted JWT for a Logto-issued one later does not touch the component or Audit's verifier. — RFC 7519: JSON Web Token

15 / Verdict

What this spike shows, and what it leaves open.

What's working
  • End-to-end round trip is fast and the rendered surface matches admin pixel-for-pixel
  • Platform owns zero data model. No JSON ingestion, no parallel ERB
  • Build pipeline is one rake task and one middleware. Heroku release does the rest
  • Auth carries a verified user identity and a role claim Audit acts on
  • Migration path to npm distribution and OAuth Token Exchange stays open
Worth deciding
  • When the third component arrives, decide between inline-CSS-per-bundle and a single shared stylesheet
  • When the contract stabilizes, decide between mutable /v1/ and an npm-distributed package
  • If a third sibling app shows up, move from pairwise shared secrets to a Logto-mediated token exchange
  • Production rollout: rotate the shared secret out of dev fallback into Rails credentials proper
End

A small tag. A clear line.

Platform got a surface. Audit kept everything that makes it ours.
