02 · Two gates, both ours

Most groups never see a human. The ones that do route to two distinct gates, and we own both halves.

Stage 1 · Internal review

Audit-side decision

Below-threshold monthlies route to Jill. She approves as-is or asks for more audits. Initials always auto-approve here regardless of score, by product call.

  • internal_approved_at + internal_approved_by_id
  • internal_more_audits_requested_at
  • concluded_at (cron-stamped, under-target)
Stage 2 · Release gate

Platform-side decision

Above-threshold groups bypass the gate entirely. Below-threshold groups (monthly or initial) hold for explicit external approval by Brad and Brian on Platform.

  • auto_released_at (system bypass)
  • external_approved_at + external_reviewer_id
  • external_rejected_at + external_reviewer_id

03 · The unit

AuditProgramCycle. One row per coder's body of work in a program for a given period.

One row per (audit_program, coder, period_key). Materializes on the first evaluation.completed event for the group and accumulates join rows as evaluations complete.

  • ▸ Monthly programs: period_key like "2026-04", one row per coder per month.
  • ▸ Initial round 1 / round 2: period_key is nil, exactly one row per coder per program.
  • ▸ Direct programs and one-off programs: not released, no approval_config, no cycle rows.

UI vocabulary stays specific to Jill's case ("Approve" button, /audits/approvals route). The underlying record is honest about being broader.

05 · Live versus released

Live aggregate cached on the cycle. Released roster pinned via released_at on the join. Per-evaluation scores are immutable post-completion, so release-time aggregates re-derive deterministically.

Live cache
aggregate_score
Decimal column on the cycle. Recomputed from source on every relevant evaluation event. Reflects active completed evaluations right now. Keeps moving as evaluations complete, get discarded, or get resurrected through revisions.
RECOMPUTED ON: evaluation.completed, evaluation.discarded
Released roster (no cached scalar)
released_at on join · released_overall_score / released_categorical_scores / released_evaluation_scores
No snapshot column. auto_release! and external_approve! stamp released_at on the cycle-evaluation joins whose Evaluation is currently active. Released aggregate, categorical, and per-evaluation scores are instance-memoized methods that read the released set on demand. external_reject! stamps nothing. Revoke clears, re-issue re-stamps.
STAMPED ON: auto_released_at, external_approved_at · DERIVED FROM: released_evaluations

06 · Per-cycle target override

target_count_override lifts the target for one cycle's period.

A nullable integer column on the cycle row. The override lives where it belongs: on the cycle itself, naturally bounded to period_key. Next month's cycle is a fresh row with no override and falls back to the membership default. April's bump never reaches May.

effective_target_count on the cycle returns target_count_override || membership.audit_target_count. target_met? and under_target? ask whether evaluations_count >= effective_target_count.

Trigger Quota learns the same lookup. When the nightly trigger runs AuditTriggers::Queries::Quota on a monthly-cadence program, it reads target_count_override for the same (program, coder, period_key) tuple and uses it when present. The nightly fills remaining slots after a partial Add More.

Three classes. One conditional family per file. Sub-services are pure decisions, the orchestrator owns the writes.

08 · Per-completion materialization

M2.5 · ResolutionHandler subscribes to evaluation events. The job recomputes and resolves.

01 SUBSCRIBE on evaluation.completed and evaluation.discarded, enqueue ResolutionHandlerJob.
02 SKIP / FIND Skip if program has no approval_config. Find or create the cycle and the join row.
03 RECOMPUTE Query fresh state, rewrite evaluations_count and aggregate_score on the cycle. Never delta-track.
04 RESOLVE Call Resolver.call(cycle:). Both gates run in one pass when possible.

Recompute, never delta-track. EvaluationDiscardForm can resurrect an original on revision-discard without firing a new event. Full recompute on every event reflects the resurrected state without special-case code.

09 · Daily under-target cron

M2.6 · Catches monthlies whose period elapsed without reaching target.

Five audits sampled, only three completed by month-end because of chart availability or late auditor work. The per-completion edge will not fire again. The cron picks them up.

01 SCAN Find monthlies past their period, under target, all attached audits' evaluations completed, Gate 1 not yet settled.
02 CONCLUDE Call cycle.conclude!. Stamps concluded_at, emits audit_program_cycle.concluded.
03 RESOLVE Call Resolver.call(cycle:). The orchestrator now treats the concluded cycle as eligible. Same path as the listener.

Initials never get cron-finalized. Under-target initials sit indefinitely. Explicit product call.

10 · Eventing

Decision moments are events. Live caches are not.

  • audit_program_cycle.more_audits_requested
  • audit_program_cycle.auto_released
  • audit_program_cycle.externally_approved
  • audit_program_cycle.externally_rejected
  • audit_program_cycle.concluded

Jobs emit on rescue

  • audit_program_cycle.resolution_failed
  • audit_program_cycle.rollover_failed

Convention

actor_id auto-injected from Current.user&.id. Auto-path events firing from background jobs naturally have nil actor_id, which is the correct signal that no human took the action.

No klient_visible event. Subscribers compose by registering against both auto_released and externally_approved.

11 · approval_config

A program has approval_config or it doesn't. That's the gate.

Programs without approval_config are excluded from the workflow. This replaces the hardcoded "direct programs excluded" rule and lets monthly / initial / one-off / direct become config differences rather than code branches.

FieldTypePurpose
periodsymbol:none for initials, :monthly for monthlies
requires_manual_review_below_thresholdbooleanGates monthly-below-threshold routing to Jill
add_more_countintegerDefault 5. How many additional audits per Add More click.

Provisioner stamps the canonical config at platform-source program creation time. M1.3 backfills existing platform programs.

13 · M3.2 · AddMoreAudits

One transaction. Five steps. The bump is per-period, never permanent.

01 READ Read add_more_count from the program's approval_config (default 5).
02 OVERRIDE Write cycle.target_count_override = effective_target_count + add_more_count.
03 STAMP Call cycle.request_more_audits!(by:). Stamps the timestamp and by_id, emits the event.
04 QUERY Short inline AR query. Chart.without_audits + filters + order("RANDOM()").limit(N).
05 CREATE For each picked chart, call Audits::Creator.call(...). Atomic with the override write and the cycle stamp.

No charts available is success with audits_created: 0. The override persists; nightly fills the remaining slots if charts arrive before period close.

15 · M5 · Platform integration

A spike. The shape of the Platform handoff is what M5 figures out.

M5 is a single ticket. Output is a decision document plus an open-questions list reviewed with Platform. Optional local prototype if a clear lead emerges. Real implementation tickets get drafted after the spike lands.

Shape A
JSON endpoint
Mirrors /api/v1/ conventions, X-API-Key, Pagy, Pundit. Optional write endpoint that PATCHes Platform's release decision back.
Familiar territory

Turbo frames
Platform pages embed turbo frames fetching rendered fragments from Audit. Less duplicate UI work for Platform. Auth and CSP are open.
Tighter coupling

Web components
Reusable custom element shipped from Audit, embedded in Platform. Larger upfront cost. Build pipeline and versioning are open.
