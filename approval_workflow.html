Approval Workflow

Game plan for the upstream gate that ships completed audit programs to Platform. Working synthesis, open questions, push architecture, issue breakdown.

Audit Team Capital Expense

01 · The problem

Audit results go out by hand. Nobody is tracking the trip.

Today, results are reviewed manually, exported to PDF or Excel, and emailed to Klients. The system has no record of whether that has happened. Below-95% monthly programs have no quality gate before they ship. Operational drag, error risk, and zero auditability around “is this done yet?”

A second motivation worth naming: 5 monthly audits is sometimes too few. One bad audit can skew the aggregate score, especially for inpatient. The “Add 5 more” mechanic is intentional sample-size protection.

02 · The vision

Eventually: auto-release to Klient dashboards. This project is the upstream gate.

Long term, audit results land automatically in Platform-side Klient dashboards. Brad and Brian own the final approval inside Platform before Klients see anything. This project sits one step upstream: the place where Jill reviews below-95% monthlies and decides whether to add more samples or release the result for Platform to consume.

“Approval” in this app means make data available to Platform. It is not approval-for-reporting. The existing internal Reporting page (PDF/Excel exports) stays open as today.

03 · Scope

What we are building, and what we are not.

In scope

  • Single Monthly tab in Jill's queue
  • Below-95% monthlies appear for review
  • Detail page with problem audits only
  • Approve action, Add 5 More action
  • Internal note per program
  • Approvals (history) tab for resolved problem children

Out of scope

  • Initials tab / approval flow
  • A new "lead auditor" role
  • Coder education / coaching dashboards
  • Program-progression dashboards
  • Klient or Koder facing views
  • APIs feeding info to Platform

04 · Audience

Just Jill. Same Pundit policies that already exist.

Permissions extend to anyone with the system_admin or kode_auditor_manager role. No new role needs to exist. Existing access policies cover it. Brad and Brian's downstream approval lives in Platform and is out of scope here.

05 · How a release happens

Four release paths converge on the same record.

Every audit program, when complete, produces a release to Platform. How it got triggered varies. The resulting record is the same shape.

01

Manual approval

Below-95% monthly. Jill clicks "Approve" in her queue.

reason: manual_approval
02

Auto above-threshold

Monthly completes with score ≥ 95%. Skips Jill's queue entirely.

reason: auto_above_threshold
03

Auto-initial

Initial round 1 or 2 completes (10 audits with submitted evaluations) regardless of score.

reason: auto_initial
04

Auto post-add-more

Jill picked "Add 5 More" earlier. New audits complete and the release fires automatically with whatever score it lands at.

reason: auto_post_add_more

06 · State machine

A release row never goes back. Three states, terminal end.

pending_review
Below-95% monthly has completed. Row exists, awaiting Jill's decision. released_at is null. Note is editable.
awaiting_more_audits
Jill picked "Add 5 More". Target bumped 5→10. New audits in flight. Row stays alive while they complete. No second decision once chosen.
released
Terminal. released_at set, reason stamped, push to Platform queued. Row appears in Approvals history if reason was manual_approval.

No transitions return to pending_review. Once Jill chooses, that's binding. Add 5 More is one-shot per monthly.

07 · Why "Release" not "Approval" suggestion

Most paths are not approvals. My read is the model name should reflect that.

Manual approval of a below-95% monthly is the rare special case. Above-95% monthlies and all initials auto-release without anyone clicking "Approve". Naming the underlying model AuditApproval would mis-name the common case.

My suggestion: name the domain model after what it represents (data being released to Platform), not how it got there. Something like AuditProgramRelease. The UI vocabulary stays specific to Jill's case. Her button still says "Approve", her page still lives at /audits/approvals. The underlying record is honest about being broader. Open to other names.

09 · The Add 5 More mechanic

Bump the cap, sample now, fill organically.

When Jill clicks Add 5 More, we bump the membership's audit_target_count from 5 to 10 and immediately invoke a one-shot sample/create pass for that coder + program. Whatever eligible charts already exist get audits right away. Subsequent nightly trigger runs continue filling any open slots as new charts arrive through ingestion.

Constraint: the new audits must be sampled from charts coded within the same program month. The program timeline does not extend. Unfilled slots at month-end are silently dropped. Once-per-monthly only. You cannot keep adding 5 more after that.

10 · Audit → Platform handoff

Push wins on every operational criterion.

Two architecturally different ways to get release data into Platform's hands: we push when we approve, or Platform pulls from us when it needs to render dashboards. Comparison:

Criterion Push (recommended) Pull
Symmetry with current Platform→Audit traffic ✓ ✗
Platform owns the write path; we don't dictate their internals ✓ ✗
Resilient if either side is briefly down ✓ (Sidekiq retries) ✗ (runtime coupling)
Klient PDF/Excel exports work without Audit running ✓ ✗
Audit stays single source of truth partial (Platform copy) ✓

11 · Push implementation

EventBus → Sidekiq job → Platform endpoint.

Reuses every existing Audit pattern: synchronous EventBus emit, async Sidekiq job for the HTTP call, retries with exponential backoff, Bugsnag-with-warning on retry exhaustion.

# When a release is created (manual or auto), emit on EventBus.
EventBus.emit("audit_program.released", {
  audit_program_id: program.id,
  audit_program_release_id: release.id,
  released_at: release.released_at,
  released_by_id: release.released_by_id,
  reason: release.reason
})

# Subscribers in config/initializers/event_bus.rb
on "audit_program.released", EventWriter, :on_event
on "audit_program.released", AuditProgramReleases::PlatformPushHandler, :on_released

# Sidekiq job calls Platform with idempotent payload.
# Gated by a feature switch so the push is a no-op until Platform is ready.
class AuditProgramReleases::SendToPlatformJob
  include Sidekiq::Job
  sidekiq_options retry: 5, queue: :default
end

Platform won't be ready to consume on day one. Build the full pipeline, leave the outbound HTTP call gated by a feature switch (env var or whatever pattern matches our app's conventions). When Platform turns the switch on, no code change required on our side. TODO comments inside the client/job file to keep this discoverable.

12 · Payload shape starting point

Translate to a flat-ish "release" Platform can land cleanly.

Platform doesn't model audit_program or evaluation the way we do. Send a simple envelope keyed on our program id for idempotency, with the audits flattened underneath. As for how much data: open question. The shape below assumes "what our reporting page renders" is enough. Could be too thin if Platform needs full line-item detail for PDF/Excel exports, could be more than they want for dashboards. To be confirmed with the Platform team.

{
  "release": {
    "external_program_id": "ap_12345",
    "kind": "monthly | initial_round_1 | initial_round_2",
    "client_id": 123,
    "coder_id": 456,
    "period": { "year": 2026, "month": 4 },
    "released_at": "2026-04-29T...",
    "released_by": { "auto": false, "user_id": 789 },
    "aggregate_score": 92.5,
    "reason": "manual_approval",
    "internal_note": "...",
    "audits": [ /* per-audit summary, expandable */ ]
  }
}

14 · Open questions

Decisions still pending. Framed as proposals, not decrees.

A. Platform adds an inbound endpoint we POST to (recommended). B. We expose API endpoints and Platform pulls. C. Hybrid (push notification, pull details).

Queue materialization
A. Queue is a SQL query, release row created at approval time. B. Pre-create a pending row when the program enters eligibility, queue is where(state: pending_review). B gives cleaner state transitions and lets Jill's note exist before she approves.
B
Payload detail level
A. Summary equivalent to our reporting page (aggregate score, categorical scores, audit list). B. Full evaluation comparisons row-by-row. C. Summary now, separate detail endpoint later if Platform needs it.
C · keeps payload small, defers complexity
UI naming for the released state
A. "Approved". B. "Released". C. "Sent to Platform". Button still says "Approve" either way. The question is what the resulting state reads as in the table.
A or B · low-stakes
Score math edges
Aggregate uses non-discarded evaluations. For revisions: use the revision's score (not the discarded original). Worth confirming before snapshotting.
use revisions, ignore discarded
Approvals history scope
A. Manual approvals only. B. Manual approvals + auto-post-add-more (anything that ever entered Jill's queue and got resolved). C. All releases regardless of how triggered. Argument for B: history is for "problem children that got resolved," and Add 5 More cases qualify.
B
Platform consumption switch (mechanism)
The switch itself is decided (build the pipeline, gate the outbound HTTP call until Platform is ready). Open: which mechanism? A. Env var. B. Database-backed feature flag. C. Whatever the app already uses.
