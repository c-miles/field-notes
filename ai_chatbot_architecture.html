Audit Analytics Assistant

A read-first chat agent, hosted on Amazon Bedrock for HIPAA, that answers workload and productivity questions by calling curated Rails tools, reasoning over what they return, and eventually drafting reassignments behind a human confirm. It reuses the productivity dashboard's query objects as its tools. The model drives the loop: it decides which tools to call, does its own math on the numbers, and picks the next step.

01 The loop

A hand rolled agentic loop inside the Rails monolith, the Shopify Sidekick shape. No orchestration framework. The model calls tools, reads results, reasons, and calls again until it can answer.

01

User asks

An admin types a question in the chat panel. The turn is persisted and a background job picks it up.

02

Job runs the loop

A Sidekiq job runs the agent off the web thread so long multi call turns never block a request.

03

Model decides

Claude on Bedrock either answers directly or emits one or more tool calls for the facts it needs.

04

Tools run Rails side

Each tool runs a Pundit scoped, read only query and returns named metrics, not prose, not raw rows.

05

Model reasons

It does the arithmetic, ranks, compares periods, and decides whether it needs another call or is done.

06

Answer streams back

The composed answer streams token by token into the chat via Turbo Streams.

Steps 03 to 05 cycle until the model has enough to answer, capped by a max iteration limit.
Shopify trick · JIT instructions

Each tool's usage rules ride back attached to that tool's result, only when the tool is called, instead of bloating the system prompt. Keeps the prompt lean and cache friendly as the tool set grows.

The model owns reasoning

Tools return facts. The model decides which tools, in what order, does the math on the returned numbers, and composes the reply. No business logic is hidden in prompt strings.

Timing

The system prompt injects today's date in Eastern and the active business week. The model resolves relative phrases like "this week" or "last month" itself, and tools take explicit period ranges.

02 The stack

Everything runs inside the AWS boundary. Claude is reached through Bedrock, so prompts and tool results never leave the customer account.

03 Tool catalog

Curated, named metric tools, not raw SQL. Nineteen to start, grouped by intent. Read tools are safe to call freely. Write tools are gated behind a human confirm and ship in phase three.

Workload read

team_workload_summary(period)Open counts and projected hours for every auditor.
auditor_workload(auditor, period)One auditor's open audits and projected hours.
open_audits_by_type(auditor)Open counts split by round, Initial 1, Initial 2, Monthly.
audits_awaiting_assignment()The unassigned queue, by chart kind and client.
aging_open_audits(auditor)Open audits ranked by how long they have sat.

Productivity read

team_productivity_ranking(period)Auditors ranked by completed output and productive hours.
auditor_productivity(auditor, period)Completed audits and productive hours for one auditor.
completed_audits_by_type(auditor, period)Completed counts split by round and month.
period_comparison(auditor, period_a, period_b)Trend between two windows for one auditor.
chart_kind_volume(period)Audit volume by chart kind over a window.

Qualification & roster read

eligible_auditors(chart_kind, client)Who is qualified, via AuditAccess. Powers reassignment.
auditor_specialties(auditor)The chart kinds an auditor is qualified for.
auditor_roster(client)The auditors working a given client.

Reference read

chart_kind_effort_minutes(chart_kind)The tunable minutes per audit that drive the hours math.
audit_detail(encounter_or_id)A single audit's status, coder, chart kind, dates.
coder_audit_history(coder)Recent audit activity for one coder.

Reassignment write · confirm

draft_reassignment(audit, to_auditor)Stages a move. Nothing changes yet.
confirm_reassignment(draft_id)Promotes the draft after a human clicks confirm.
cancel_reassignment(draft_id)Discards a staged move.
Guardrails

Pundit scoping inside every tool, a read only connection for reads, all tool output treated as untrusted so no raw PHI rows are returned, and a human confirm before any write. A max iteration cap bounds the loop.

Why curated, not text to SQL

Text to SQL falls off a cliff on real schemas and database over MCP against production is unsafe. Curated tools returning named metrics keep authorization deterministic and outputs correct.

From Kody v1

Keep the registry and per tool auth split and the draft then confirm handshake. Fix the two flaws, one tool call per turn and the mangled multi turn tool history, by using RubyLLM's real loop and Postgres transcripts.
