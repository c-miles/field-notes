A Walkthrough, Not a Prescription

How we background our exports.

A tour through the pieces we landed on for long-running file generation. A persisted report, a background job, a live channel back to the browser. Sharing it in case any of the shape is useful.

Audit Engineering
Background jobs, WebSockets, S3
Chris Miles
Point an agent at it, or just read it

The wall

Thirty seconds is not a lot of seconds.

Heroku routes kill any HTTP request that has not responded in thirty seconds. That is fine for page loads. It is brutal for exports. A modest CSV fits, a real one does not, and the moment the dataset grows or the user picks a wider filter, the request crosses the line.

The symptoms are familiar. A spinner, then an error page. Sometimes a half built file. Sometimes a retry that lands on a fresher dataset and produces a different answer. The user never quite knows what they got.

What goes wrong in-request
  1. 01 User clicks Export. The browser starts waiting.
  2. 02 The web worker is pinned to that one request. No other work runs on it.
  3. 03 Seconds pile up. Queries run, rows stream, memory climbs.
  4. 04 Thirty seconds. The router returns H12. No file.

The move

Separate the click from the work.

The HTTP request has one job: record that the user asked, and hand them a receipt. Everything slow, every query, every render, every upload, happens somewhere else. A live channel carries the answer back when it is ready.

01 / Persist
A Report record, the instant they click.
A tiny row gets created and saved in milliseconds. It captures who asked, what kind of export, and a snapshot of their filters. That is the receipt.
02 / Background
A Sidekiq job runs it out of process.
No HTTP timeout. No tied up web worker. The job queries, renders, and produces the file on its own time. It can retry when infrastructure blips.
03 / Broadcast
A WebSocket tells the browser when it's ready.
The moment the file is attached, a Turbo Stream pushes the row update over ActionCable. The status flips. The download button appears. No refresh.
The user sees an immediate "we've got it, we'll let you know" and keeps working. The ugly work is elsewhere. When it finishes, the UI updates itself.

The pipeline

Three zones. One story.

Web request < 1s
  • User clicks Export.
  • Controller creates a Report row and snapshots filters.
  • An event is emitted on the event bus after commit.
  • Redirect with a flash: "Your report is being prepared."
Background work seconds to minutes
  • Sidekiq picks up the job from the queue.
  • A generator service builds the file. PDF, XLSX, or CSV.
  • Active Storage attaches the bytes. S3 holds the blob.
  • Another event fires: report.completed.
Live update instant
  • A broadcast handler catches the event.
  • Turbo Stream replaces the table row over ActionCable.
  • Status flips to Completed, Download appears.
  • User never reloaded the page.
Each zone has its own budget. The web request answers fast. The job takes as long as it needs. The live channel delivers the punchline.

The receipt

One row holds the whole story.

The Report model is small, deliberate, and does all the heavy lifting. It is the durable identity of an export, the thing the job works on, the thing the view subscribes to, the thing the user downloads.

  • user, kind, name. Who asked, what flavor, how it is labeled.
  • filters. A JSONB snapshot of the state at request time, so the result never drifts from what was asked.
  • processing_at, completed_at, failed_at. Three timestamps carry the whole lifecycle. Status is derived, never stored.
  • has_one_attached :file. Active Storage owns the blob. The record owns the metadata.
Why snapshot the filters?

A report requested today should return today's answer, even if a coworker edits a default filter tomorrow, or a coder is reassigned next week.

Because the filter payload is captured on the record, the job is deterministic: same inputs, same output. The report is also regenerable on demand if anything goes wrong.

The record is the contract.

Lifecycle

Four states, derived from timestamps.

No state column, no enum to keep in sync. The presence or absence of three timestamps tells the whole story, and the UI pill reads them the same way the job writes them.

Pending
no timestamps set yet
job is enqueued
Processing
processing_at set
worker is on it
Completed
completed_at set
file is attached
Failed
failed_at set
reason recorded
A simple invariant keeps this honest: a report cannot be both completed and failed. Every transition writes exactly the timestamp it is allowed to.

The background job

Slow work, patient retries, clean failures.

The job is the only place the export actually gets built. It loads the report, stamps it as processing, asks a generator service for a file, attaches the result, and emits an event. Errors are sorted into two buckets.

  • Retryable. Credentials missing, remote session could not start, the render pipeline had a hiccup. Sidekiq retries with backoff.
  • Terminal. Bad filter, missing coder, unsupported kind. The job marks the report failed and records a reason. No retries.
  • Monadic results. Generators return a Success or a Failure. The job does not have to know which generator ran or how it works inside.
Isolation
A job, not a request.
Runs on a Sidekiq worker, not a web dyno. No HTTP timeout. A five minute render is a five minute render and nobody cares.
Containment
Retries absorb the infrastructure.
S3 throttle, Browserbase hiccup, transient network? The job tries again. The user never sees it. Only exhausted retries promote to a real failure.
Audit trail
Every transition emits an event.
report.requested, report.completed, report.failed, report.downloaded. The event log is the source of truth for who exported what, and when.

Live channel

One line on the page. No polling.

app/views/reports/index.html.erb ERB 
<%= turbo_stream_from(Current.user, "reports") %>

That single line subscribes the browser to a channel scoped to the current user. When the job finishes and emits report.completed, a broadcast handler renders the updated row partial and pushes it over ActionCable with broadcast_replace_to. Turbo Streams swaps the <tr> in place.

Status flips without a refresh. The pill changes from Processing to Completed, the Download button fades in, the timestamp updates. The user does not have to poll, reload, or navigate. They could be on another tab and come back to a finished report already sitting there.

Storage

Files live on S3, not on the dyno.

Active Storage handles the upload. The bytes go to a private S3 bucket, the Report keeps a foreign key to the blob, and the controller hands the user a short lived, presigned download URL.

  • Private by default. Nothing is public. Downloads are gated by policy and pundit scope.
  • Short lived URLs. One minute. Long enough to click, short enough to be uncopyable.
  • Decoupled from the web tier. A browser downloads from S3, not through the Rails server. The dyno stays free.
Retention
Ninety day lifespan.

A nightly cleanup job destroys reports older than the retention window. Active Storage automatically purges the attached blob when the record is destroyed, so S3 stays tidy and costs stay predictable.

Configurable

The window is driven by an env var, so retention can be tightened or loosened per environment without a deploy.

User surface

Every export, on one page. Forever.

Users never have to re-run an export to get it back. A dedicated Reports tab shows every report they have ever requested, live status pills, filter summaries for context, and a Download button the moment a report is ready.

Without a Reports tab
Run it, or lose it.
If the user closes the tab, they start over. If support gets a "can you pull that again?" ticket, an engineer reruns the query. The export is ephemeral. Every download is a fresh five minute wait.
With a Reports tab
History with a download button.
Past exports are right there, sorted by recency. Completed reports are one click away for the life of the retention window. Failures are labeled with a reason so the user knows to try again, not "why is this broken?"

Why it wins

Six benefits, one architecture.

✓
No more timeouts
The HTTP response is a receipt. The work that used to hit the wall at thirty seconds now runs on a worker with no wall.
✓
Users keep working
They click, they get a confirmation, they go do something else. The export follows them back when it is ready.
✓
Live feedback, no polling
A WebSocket carries status and a download link straight to the open page. No timers, no refresh buttons, no "is it done yet?"
✓
Re-downloadable for ninety days
Support tickets about "can you pull that again?" stop reaching engineering. Users self-serve from their own history.
✓
Resilient by default
A flaky S3 or a browser render hiccup gets retried by Sidekiq, not surfaced as a user-facing error. Only real failures reach the human.
✓
Every export is auditable
Events flow through one bus. Who asked, when it ran, whether it succeeded, who downloaded. The whole life of a file is logged.

Where to look

The pieces, by folder.

Nothing exotic. A model, a service, a job, a handler, a channel, a retention task. Each piece has a single responsibility and composes with the rest.
