Filter Persistence · Server-Side Move

Filters that remember.
And don't flash.

The flash

Default-painted. Then yanked.

Filter state lived in localStorage under keys like filterMemory:/audits. On connect(), a Stimulus controller read the saved query string and called Turbo.visit(savedUrl, { action: "replace" }). The server already rendered the default-state filter form by then, so the user saw defaults paint, then a re-render snapped to their saved state.

The flash was structural. Any JS-based restoration runs after the first paint by definition.

Before, each visit
  1. 01 Server renders filter form with defaults (status=open, auditor=me).
  2. 02 Browser paints. User sees default state.
  3. 03 Stimulus connects, reads localStorage, triggers a Turbo visit.
  4. 04 Flash. Page re-renders with saved state.

The move

Push persistence under the render.

01 / Store
One jsonb row per user, per page.
A new user_filter_preferences table keyed on (user_id, page_key). Durable, queryable, survives browser resets and follows the user across devices.
02 / Restore
Before the view renders.
A controller concern hydrates filter params from storage on a bare visit, so the first HTML the server emits already carries the user's saved values. No JS round-trip.
03 / Write
Only on deliberate edits.
A signal from autosave gates the write, so link clicks and server redirects stop overwriting preferences that the user never asked to change.
First paint is correct. No flash, no second trip, no client-side state owning the source of truth.

Schema

One row. One composite index.

db/migrate/..._create_user_filter_preferences.rb Ruby 
create_table :user_filter_preferences do |t|
  t.references :user,  null: false, foreign_key: true, index: false
  t.string     :page_key, null: false
  t.jsonb      :filters,  null: false, default: {}

  t.timestamps
end

add_index :user_filter_preferences, [:user_id, :page_key], unique: true

page_key mirrors Rails controller_path, so ten of eleven filter pages wire up with zero per-controller mapping. The unique composite prevents duplicate rows and gives the upsert a clean ON CONFLICT target.

Model API

remember, recall, forget.

app/models/user_filter_preference.rb Ruby 
def self.remember(user:, page_key:, filters:)
  filters = filters.to_h

  existing = find_by(user:, page_key:)
  return if existing.nil? && filters.blank?       # skip first-visit noise
  return existing if existing && existing.filters == filters  # no-op write

  record = existing || new(user:, page_key:)
  record.filters = filters
  record.save!
  record
rescue ActiveRecord::RecordNotUnique                     # concurrent-insert race
  record = find_by(user:, page_key:)
  record&.update!(filters:)
  record
rescue ActiveRecord::ActiveRecordError => error
  report_failure(error, user:, page_key:, operation: :remember)
  nil
end
Any storage failure logs a warning and reports to Bugsnag at warning severity. Filter persistence is a convenience; a DB blip should never take down the index page that triggered it.

Orchestrator

FilterPersistence ties storage to the filter.

app/controllers/concerns/filter_persistence.rb Ruby 
def with_filter_persistence(page_key: controller_path)
  filter = yield(effective_filter_params(page_key:))
  remember_filter_state(filter, page_key:) if autosave_submitted?
  filter
end

def autosave_submitted?
  params[:autosaving].present?
end

# Returns the request params when the URL carries filters, or a fresh
# Parameters copy with stored values layered in on a bare visit.
def effective_filter_params(page_key:)
  return params if params[:filter].present?

  stored = UserFilterPreference.recall(user: Current.user, page_key:)
  stored.is_a?(Hash) ? params.merge(filter: stored) : params
end
No param mutation. params.merge returns a new Parameters object; the request's own params stays what the user sent. The composed copy flows through the normal filter API as if it had always been there.

The factory case

One concern. Two entry points.

Most controllers build their filter with Filter.new(...) and then assign params. apply_filter_params wraps that shape.

Reporting is different. ReportingFilter.for_request is a factory that reads params during construction, so restore has to happen before the filter exists. The block form yields the effective params and lets the caller thread them wherever the factory would have received params.

app/controllers/reporting_controller.rb Ruby 
def index
  authorize :reporting

  @filter = with_filter_persistence do |effective_params|
    ReportingFilter.for_request(
      params: effective_params,
      requester: Current.user,
      allowed_sources: Current.allowed_sources
    )
  end

  # ... render ...
end

The write signal

autosaving=true. That's the line.

autosave_controller stamps autosaving=true on every request it initiates. Persistence fires only when that flag is present. The server cannot see HTTP referers reliably or tell a form submission apart from a link click through URL alone. The flag gives us an intent signal that's explicit and already wired up.

Old path (body only) body.append('autosaving', 'true')
GET carries params in the URL.
The body is null. Silent no-op.
New path (URL or body) if (body) body.append(...)
else url.searchParams.append(...)
Reaches the server on every autosave.

Back / Forward

From chip-by-chip to page-by-page.

filter_url_sync_controller used to push a history entry on every autosave flush. Hitting Back on a filtered audits page would rewind filter edits one at a time before leaving the page. Three chips added, three presses of Back to escape.

The controller is now 60 lines instead of 204. replaceState only. Every filter edit updates the URL in place, so the history stack reflects pages visited, not filter states traversed. Bookmarking still works.

Before Back undid edits. pushState per autosave flush. Three chips = three history entries. Back went through your filter history before leaving the page.
After Back leaves the page. replaceState on every submission. One history entry per page visit. Back goes where the user expects: the previous page.

The quiet Turbo gap

HTML attributes and IDL properties disagree.

Turbo's page cache snapshots the outgoing page via outerHTML, which reads attributes. When you type in an input or pick an option, the browser updates the IDL property (input.value, option.selected) but does not touch the matching attribute. The cached preview on Back reflects the attribute world; the user's edits live in the IDL world.

Attribute world (what Turbo sees) <input name="filter[encounter_number]" value="">
<option value="monthly">Monthly</option>
IDL world (what the user did) input.value = "ENC-12345"
option.selected = true
Turbo deliberately doesn't do this promotion. The Hotwired team closed the request and pointed at turbo:before-cache as the right seam. So we take the seam.

cache-safe-form

Promote IDL to attribute before the snapshot.

app/javascript/lib/form_state.js JavaScript 
export function promoteFormStateToAttributes(root) {
  root.querySelectorAll("input, textarea, option").forEach((element) => {
    switch (element.tagName) {
      case "INPUT":    promoteInput(element); break
      case "TEXTAREA": element.textContent = element.value; break
      case "OPTION":   element.toggleAttribute("selected", element.selected); break
    }
  })
}
A Stimulus controller (cache-safe-form) listens for turbo:before-cache and calls the helper against its form. Idempotent and composable with other before-cache handlers: running it twice produces the same DOM.

TomSelect, the trickster

Two extra lines that matter.

TomSelect snapshots innerHTML at init time. On destroy(), it restores that snapshot, reverting the <select> to its pre-init state and losing any user picks. The generic cache-safe-form promotion is not enough on its own.

select_controller absorbs the extra dance in its own before-cache handler. Three independent try/catch steps so a TomSelect internals throw cannot skip the visibility toggle that hides the raw native listbox during the outgoing view transition.

app/javascript/controllers/select_controller.js JavaScript 
// 1. Sync TomSelect's internal selection to the native select
ts.updateOriginalInput()

// 2. Promote IDL state to attributes (shared helper)
promoteFormStateToAttributes(this.element)

// 3. Override revertSettings so destroy() writes the live HTML
if (ts.revertSettings) {
  ts.revertSettings.innerHTML = this.element.innerHTML
}

Why it holds up

Four boundaries, drawn cleanly.

Boundary 01
Non-mutation.
The concern composes a new Parameters object instead of rewriting params. Downstream code sees the honest request; the merged copy flows only through the intended callsite.
Boundary 02
Signal-gated writes.
One flag, one rule: deliberate submission writes, everything else displays. The rule is enforced at the concern's single exit point, not scattered across controllers.
Boundary 03
Defense in depth.
URL-only fields like audit_trigger_execution_id are not in the filter form, so they can't be in an autosave submission, so they can't be persisted. The trap resolves at the right layer.
  • Three independent controllers on every filter form. autosave persists, filter-url-sync keeps URLs clean, cache-safe-form protects the Turbo snapshot. Idempotent, composable, ordering-independent.
  • Failure modes degrade. A Bugsnag-reported storage error logs, reports at warning severity, and returns nil. The index page keeps rendering.

Filter state lives on the server now. The first paint is correct. Back works. Links stay links. The cached preview stops lying.
